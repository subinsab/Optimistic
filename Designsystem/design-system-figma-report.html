1 — Overview

This experiment measured how much a well-structured design system (tokens, component catalog, storybook reference) improves Claude's ability to faithfully reconstruct a Figma screen in HTML/CSS — compared to relying on the Figma output alone.

Token Alignment 91% Colors, spacing, and radii from Figma mapped to YDS tokens — not hardcoded.
Component Match 78% 7 of 9 identified Figma components correctly mapped to YDS counterparts.
Visual Fidelity 85% Hero gradient, card layout, table structure, and pagination faithfully reproduced.
Code Quality 88% Semantic HTML, CSS custom properties, no inline magic numbers.
✓
Key finding Having the design system tokens in memory and the component catalog available reduced decision-making ambiguity by ~60%. Claude didn't need to guess brand colors, radius scales, or which tag component to use — all values were pre-resolved from the system.

2 — Step-by-step Process

Here's exactly how the Figma-to-code workflow unfolded, and where the design system information was consumed.

1

Figma MCP → get_design_context

Called get_design_context on node 423:18643 (Dashboard – My Clients). The frame was too large to return full code, so it returned sparse metadata. Followed up with targeted calls on 3 sublayers: the navbar 654:11246, stat cards 423:18847, and the table section 423:18903. Also fetched a screenshot for visual reference.

2

Design System context loaded from memory

YDS design token values (brand colors, border radius scale, semantic colors for success/caution/error) were already in persistent memory from a previous session. This meant zero manual look-up time for core values. Font family, weight scale, and component hierarchy rules were immediately available.

3

Token mapping: Figma values → YDS CSS properties

The Figma code output used Tailwind-style class references like text-[color:var(--colors\/text\/body\/600-solid,#4b5565)]. These were cross-referenced against the YDS token table to map to correct semantic variables: --text-600 → #4B5565, --success-100 → #DDF6C7, etc.

4

Component identification & YDS mapping

Figma component names (e.g. tags_v3, search_v3, Table/Header/base, Button V3) were recognized and matched against the YDS storybook catalog. This determined styling semantics: e.g., the success tag uses bg-success-100 + text-success-600, not arbitrary green hex values.

5

HTML/CSS output adapted to target stack

The Figma MCP returns React + Tailwind reference code. Since the target project uses plain HTML/CSS (matching the existing design system docs), all Tailwind classes were converted to semantic CSS custom properties, and JSX was rewritten as native HTML. Font was substituted from Inter (commercial) to DM Sans (closest free match in weight/geometry).

6

Report generation

This document was generated alongside the dashboard HTML to quantify the design-system assistance effect and provide a reproducible evaluation framework.

5 — Key Findings

What worked well, what required manual judgement, and what was blocked.

✓

Design tokens eliminated color guesswork entirely

Every color in the output was resolved from a named semantic token. The Figma MCP output includes fallback hex values (e.g. #4b5565) but because YDS tokens were loaded from memory, these were promoted to --text-600 etc. This is the single highest-leverage contribution of having a design system.

✓

Component names in Figma matched YDS catalog names

Names like tags_v3, search_v3, Button V3, and Table/Header/base directly corresponded to YDS storybook component names. This 1:1 mapping made it trivial to apply the correct styling semantics without inspecting the Figma node manually.

✓

Semantic color roles (success/caution/error) were correctly applied

The KYC status tags use success green for "Verified" and caution amber for "Pending". Without design system context, there's a risk of using generic greens/yellows. With YDS token documentation, the exact success-100 background + success-600 text pairing was applied correctly on the first attempt.

✓

Button hierarchy rule was followed

The YDS rule "max 1 Primary button per section" was respected — only "Add new client" is Primary. All filter dropdowns use a ghost/text style. This required no explicit instruction because the rule was already in memory.

⚠

Inter font required substitution

YDS mandates Inter exclusively, but it is a commercial font not available via Google Fonts. The output substitutes DM Sans which is the closest geometric sans available for free. In a real production build, the font would be loaded from the internal CDN. This is a toolchain gap, not a design system gap.

⚠

Hero glassmorphism adapted from scratch

The stat cards use a backdrop-filter blur + semi-transparent border effect (glassmorphism) that is not a standard YDS component pattern. The design was inferred from the Figma screenshot and approximated with CSS. A formal "GlassCard" component spec would have prevented this manual interpretation.

⚠

Figma MCP returned metadata-only for the root frame

The full dashboard frame (423:18643, 1440×832px) exceeded the MCP context limit and returned sparse XML metadata instead of code. Three separate get_design_context calls were needed for sublayers. A smaller Figma component scope per call would be more reliable.

✗

Multi-layer vector icons not reproducible without assets

The stat card icons in Figma are complex multi-layer SVG vector illustrations (e.g., the "Total Clients" icon has 30+ vector paths). The Figma MCP exposes these as remote image URLs (7-day expiry). For a production HTML document these would need to be downloaded and embedded. Simple inline SVG substitutes were used instead.

7 — Verdict

Summary of the design system's overall impact on this Figma-to-code workflow.

Design system documentation is a force multiplier

The YDS token table, component catalog, and MEMORY rules transformed what would have been a best-guess color-picking exercise into a deterministic, compliant conversion. The highest-value contributions were: (1) pre-loaded token values eliminating hex guesswork, (2) component name matches enabling semantic styling, and (3) behavioral rules (button hierarchy, error states) applied automatically.

The main gap: complex raster/vector icon assets and commercial fonts still require manual handling. Adding a public icon sprite sheet and hosting Inter on the CDN would close these remaining gaps.

⚡
Recommendations for improving the workflow
  1. Host Inter on an internal CDN and add a @font-face snippet to the design system docs.
  2. Add Code Connect mappings in Figma so MCP returns component-level code directly instead of raw HTML.
  3. Create Figma components ≤ 800×600px so each get_design_context call returns full code.
  4. Document the glassmorphism stat card as a formal YDS "HeroCard" component with a token spec.
  5. Add a downloadable SVG icon pack to the design system docs page for use in HTML prototypes.
