First message — gives Claude full system context
I've attached yubi-market-design-system.md — the complete Yubi Market design system reference.

Please read it fully and confirm you understand the following before we start building:

1. TYPEFACE: Inter only. Weights 400 and 700 only. Never Inter, Roboto, or system fonts.
2. COLORS: Always use semantic tokens (e.g. color.text.primary = #121926). Never raw hex in components.
3. SPACING: 8px base unit. Only use space.* tokens (space.050=4px, space.100=8px, space.200=16px, space.300=24px).
4. RADIUS: Buttons and inputs = radius.md (6px). Cards = radius.xl (12px). Pills = radius.full (9999px).
5. SHADOWS: Cards = shadow.sm. Dropdowns = shadow.md. Modals = shadow.lg.
6. FOCUS RINGS: Always border-color: #81AFF2 with box-shadow: 0 0 0 3px rgba(129,175,242,.45).
7. DARK MODE: Use [data-theme="dark"] overrides from the file — never hardcode dark values.
8. FEEDBACK: Always apply full sets — bg + border + icon + text. Never a single feedback color alone.

When generating UI, output clean semantic HTML with inline or scoped CSS using the exact tokens from this file. Do not invent new tokens. Do not use utility class frameworks. Match the Yubi Market visual language precisely.

Confirm you are ready.

Guide — Writing PRDs that work with Claude

Name components explicitly

Always use the exact component name from the system. Write "use a Drawer (right, 480px)" not "a side panel thing". Claude maps component names directly to token sets and layout rules.

DO Status column: Pill badge — radius.full, feedback success set
DON'T A little green dot or label showing status

Specify all four states

Every section of a screen can be empty, loading, populated, or in error. Claude will default to the happy path only unless you name every state explicitly in section 4 of the PRD.

DO Loading: skeleton rows (radius.sm, color.bg.subtle, animated pulse). Empty: centered H4 + B3 + primary CTA.
DON'T Handle edge cases too

Declare the layout template

Claude needs to know the shell before it can build the content. Start every PRD with the full layout: which sidebar pattern, topbar height, content area margins, and grid column count.

DO Layout: fixed sidebar 272px + 56px topbar + 12-col grid, 56px content margin, max-width 1280px.
DON'T Standard dashboard layout

Use token names, not px values

Writing token names instead of pixel values triggers Claude to use the correct CSS variable. It also prevents hallucinated values that look similar but break the system.

DO Padding: space.300 (24px). Title: H3 token. Border radius: radius.xl.
DON'T About 20-24px padding, largish text, rounded corners

Define grid spans for every section

Tell Claude how many of the 12 desktop columns each section occupies. This prevents Claude from making layout decisions that don't align with your visual intent.

DO KPI row: 4 cards × 3 cols each. Chart: 8 cols left. Activity feed: 4 cols right.
DON'T Charts on the left, some cards on the right

One PRD per screen, not per feature

Each PRD should describe exactly one screen or one major modal/drawer. Mixing multiple screens in one PRD causes Claude to blend layout decisions and produce incoherent output.

DO PRD-01: Dashboard. PRD-02: Client Detail Drawer. PRD-03: Add Client Modal.
DON'T The whole app with all screens and modals

How to structure the .md for maximum Claude comprehension

1
Lead with hard rules

The first section of your .md must be a "Rules Claude Must Follow" block. Claude weights content at the top of documents more heavily. Put your non-negotiables there — typeface, token-only colors, no utility frameworks.

2
Tables over prose for tokens

Token tables (Token | Hex | Use for) are parsed more reliably than token definitions written as sentences. Claude can extract structured values from markdown tables without ambiguity.

3
CSS block at the end

End the .md with a single CSS block containing all custom properties. When Claude generates code, it will copy-paste variable names verbatim from this block — zero hallucination risk for token names.

4
Keep it under 8,000 tokens

Claude's effective comprehension degrades at very long documents. The yubi-market-design-system.md file is sized to stay well within this limit. Don't pad it with narrative text — only structured data.

5
Use consistent section headers

Match heading levels to logical depth. ## Color System → ### Semantic Tokens → #### Text. Claude navigates by heading hierarchy when looking up specific values.

6
Add a Quick Reference at the top

A "Key Values" table with the 12 most-used tokens near the top of the file means Claude can answer most questions without reading the full document — faster, more accurate output.

### TYPOGRAPHY RULES
1. Inter exclusively — Regular 400 and Bold 700 only. Never Inter, Roboto, system fonts.
2. No custom font sizes — always reference a named token (D1–T2)
3. Headings: line-height tight (1.15) or snug (1.25)
4. Body text: line-height normal (1.5), max 65ch line length
5. Uppercase labels must use letter-spacing: 0.04em+ and weight 700
6. Fragment Mono for code snippets, token names, hex values, numbers in tables

1. Mobile-first — write base styles for xs (320px), override upward with min-width queries

Tips for Best Results

A few habits that make Claude output dramatically more accurate and consistent.

Start every session fresh

Attach yubi-market-design-system.md and send the Setup Prompt at the top of every new Claude conversation. Token knowledge doesn't carry between sessions.

After Claude confirms, keep the conversation focused on a single screen or feature. Start a new conversation for each major screen.

Iterate, don't regenerate

Once Claude has generated a screen, ask it to refine specific sections rather than starting over. Say "update the KPI cards to use tile-brand tiles" rather than re-sending the full PRD.

If an output looks wrong, quote the specific token or component from the .md file in your correction — Claude will self-correct reliably when given the exact reference.
