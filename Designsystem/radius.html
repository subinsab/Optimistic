radius.guidelines

Usage Guidelines

Dos and don'ts for applying radius tokens across components and compositions.

Do
Match child radius to parent minus the parent's padding. If a card uses radius.xl (12px) with 4px padding, inner children should use radius.lg (8px) to stay optically flush.
Use radius.full (9999px) for pill shapes — status chips, tags, toggle tracks, and circular avatars. This future-proofs against size changes.
Apply radius.none to elements that must bleed to a container edge — full-width images inside a card, inline code blocks in a table cell, edge-aligned banners.
Keep radius consistent across the same component type. All primary buttons in an interface should share the same token — mixing radius.sm and radius.xl on buttons looks accidental.
Don't
Never use arbitrary radius values like border-radius: 10px or border-radius: 5px 3px 7px 2px. All values must map to a named scale token.
Don't apply large radius tokens (radius.2xl, radius.3xl) to small interactive elements like buttons or inputs — it creates cartoonish proportions that feel inconsistent with the system's precision.
Avoid mixing rounded and square styles within the same component group. If stat cards use radius.xl, summary cards in the same view should not use radius.none.
Don't use radius as a differentiator for component variants unless the variant spec explicitly calls for it (e.g., a pill variant of an input). Radius variation should encode purpose, not decoration.
